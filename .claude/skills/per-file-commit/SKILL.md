# Per File Commit

## What This Skill Produces

- 変更ファイルを 1 ファイル 1 コミットで整理する手順
- 各コミットメッセージがそのファイルの意図を説明している状態
- 例外コミット（複数ファイルをまとめる）を許可する条件と明示方法
- コミット完了後に未コミット差分が残っていないことの確認

## When to Use

- 実装や修正が一段落して、複数ファイルの差分をコミットしたいとき
- 変更履歴を細かく追えるように、ファイル単位でコミットしたいとき
- レビューしやすい粒度に分けて履歴を残したいとき
- `git add -A` や `git add .` で巻き込み事故を避けたいとき

## Decision Points

1. すべての変更ファイルを本当に 1 ファイルずつ分割できるか
   - 分割できるなら、ファイル単位でコミットする
   - 密結合で同時反映が必要な複数ファイルがあるなら、例外としてまとめてよい
   - 例外にする場合は、なぜ分割できないかを先に明示する

2. コミット対象に含めるべきファイルか
   - 今回のタスクに必要な差分だけをステージする
   - 生成物・build 出力・一時ファイルは通常コミットしない
   - 既存の無関係な未コミット差分は巻き込まない

3. コミット順序
   - 依存関係がある場合は、参照される側 → 参照する側の順でコミットする
   - 純粋なドキュメントや設定の追加は最初か最後にまとめる

## Procedure

1. 変更一覧を確認する
   - `git status` で変更ファイル一覧を確認する
   - `git diff` で未ステージ差分の概要を確認する
   - すでにステージ済みの差分がある場合は、今回の対象と混ざっていないかを見る

2. コミット計画を立てる
   - どのファイルをどの順でコミットするかを決める
   - 1 ファイル 1 コミットを基本にし、例外があれば理由を明示する
   - 各コミットメッセージで何を伝えるかを 1 行で言えるか確認する

3. 1 ファイルだけをステージする
   - `git add <path>` で 1 ファイルだけをステージする
   - `git add -A` や `git add .` は使わない
   - 部分的な差分のみをコミットしたい場合は `git add -p <path>` を検討する

4. ステージ内容を確認する
   - `git diff --cached` で、ステージされた差分が想定通りか確認する
   - 余計なファイルや差分が混ざっていないことを確認する

5. コミットメッセージを書いてコミットする
   - そのファイルの変更内容と意図を 1 行で説明する
   - リポジトリの既存コミットスタイル（プレフィックス、言語など）に合わせる
   - `git commit -m "<message>"` でコミットする

6. 次のファイルへ進む
   - `git status` で残りの未コミットファイルを確認する
   - 想定外の差分が現れていないか確認する
   - 1〜5 を繰り返す

7. 例外コミットを扱う
   - 複数ファイルが密結合で、分割するとビルド失敗や意味の崩壊が起きる場合は例外を許可する
   - 例外時は、どのファイル群をなぜまとめるかをコミットメッセージ本文に明示する
   - 例外コミットでも、無関係な差分は含めない

8. 最終確認を行う
   - `git status` で未コミット差分が残っていないことを確認する
   - 残っている場合は、意図したものか、次のコミット対象かを判断する
   - `git log --oneline` でコミット粒度が妥当か点検する

## Quality Checks

- 各コミットが 1 ファイルに限定されている、または例外理由が明確である
- 各コミットメッセージがそのファイルの意図を説明している
- `git add -A` / `git add .` を使っていない
- 不要な生成物や無関係な差分が含まれていない
- 最終的に未コミット差分が意図した状態になっている

## Guardrails

- 1 ファイル 1 コミットを優先するが、整合性を壊すなら例外を許可する
- 複数ファイルを同時にコミットしないと意味が崩れる場合は、例外理由を明示して確認する
- ユーザーが求めていない git 履歴改変や destructive な操作（`git reset --hard`、`git push --force` 等）はしない
- 既存の未コミット変更が今回の作業と無関係なら巻き込まない
- secrets を含むファイル（`.env`、`credentials.json` など）は警告し、ユーザー確認なくコミットしない

## Output Format

返答には必要に応じて以下を含める。

- Commit plan: どのファイルをどの順でコミットするか
- Exceptions: 1 ファイル 1 コミットを崩す必要がある箇所と理由
- Executed commits: 実行したコミットの一覧（ファイル名 + コミットメッセージ）
- Final status: 未コミット差分の有無

## Example Prompts

- `/per-file-commit 今の差分を 1 ファイルずつコミットして`
- `/per-file-commit このブランチの変更ファイルを順番に整理しながらコミットして`
- `/per-file-commit 修正が複数ファイルに散らばっているので、ファイル単位で履歴を分けたい`
